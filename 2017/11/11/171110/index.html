한 일

  1. 로그 강의
  2. 리팩토링
      1. 컨트롤러에서 뷰에 객체를 계속 넘겨주다보니 메소드 분리가 힘들었다. DTO를 만들자.
      1. getList 안 쓰는 방법-> list 가져올 객체에 메소드를 만들어서 기능을 분리.
  3. junit - 필드를 다 내부 연산으로 생성해서 테스트 코드 짜기 힘들었다.
    이번에는 테스트용 생성자나 setList를 만들어서 처리했는데 그렇게 해도 되는지 모르겠다.

배운 것 - 포비 리뷰

  1. 파라미터가 3개 이상이면 설계 잘 못 했는지 의심해보기
  2. 객체에서 getter, getList 등으로 데이터를 꺼내오는 것은 캡슐화 위반.
    (getlist의 경우 자료구조를 set으로 바꾸면 getList 사용하는 부분을 다 바꿔야 하는 문제도 있다.)
    예) Lottos 클래스의 sumMoney()에서 lotto.getCommonCount() 를 사용하는 부분이 있다.
    → 그 부분을 떼서 Lotto 클래스에 getPrize()로 만들어줌.
  3. 되도록 로컬 변수 사용. 메소드를 넘나들 때만 필드를 만듦.
    예) InputView 클래스의 Scanner 같은 경우는 필드로 만들어도 됨.
    (or 입력 받는 메소드가 분산돼 있는데, 그걸 하나로 모아서 DTO에 값 넘겨주면
    scanner를 해당 메소드의 로컬 변수로 사용해도 된다. 메소드를 넘나들 필요 없어지니까)
  4. DTO를 사용하면 View 단에 계속 값을 주고 받느라 로직 객체를 넘길 필요가 없다.
