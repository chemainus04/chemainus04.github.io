볼링 스텝1 진행 과정

  1. 필요할 것 같은 클래스를 먼저 다 만들고 그 안에 메소드 구현하려 함.
    → 클래스가 너무 많아서 메소드를 어떻게 만들지, 객체끼리 어떻게 연결할 지 갈피가 안 잡힘.
  2. BowlingGame 클래스에 기능 다 만들고 메소드 리팩토링하면서 테스트
    → 객체지향, 클래스 분리에 대해 고민하지 않고 기능 돌아가는 것에만 집중. 시간 단축됨.
  3. InputView, ResultView 분리
  4. 컨트롤러 분리를 위해 Model 클래스로 Frame 만듦
  5. Frame 에서 쓰러진 핀 개수랑 상태 표시를 저장하는 리스트 필드, 그와 관련된 메소드를 Pin 클래스로 분리
    → 클래스 먼저 나누고 구현하는 것 보다 로직 이해하기 쉽고, 시간도 단축됨.
    이때 테스트 케이스 만들어서 junit으로 테스트 했는데, 확실히 편했다. 초반에 테스트를 위해 메소드를 만들거나 세팅할 때만 좀 번거롭고, 그 뒤로 소스 변경할 때마다 테스트코드만 돌려서 에러 확인할 수 있어서 좋았다.
    • 리뷰: Pin이 프레임 돌 때마다 핀, 출력값 리스트를 추가하는데 프레임을 10개 만들고 그 안에서 쓰러진 핀 개수만 변수로 들고 있는 게 어떨까? 그럼 리스트를 만들 필요도 없어진다.
  6. 클래스 설계 변경, 그에 따라 메소드 대거 변경 + 테스트 + 메소드 쪼갬
    → 변화해서 쌓이는 값을 저장한다고 생각하면 무조건 리스트를 떠올렸던 것 같다. 프레임 객체를 10개 만들고 거기서 핀 변수를 가진다는 생각을 못 했는데, 좀 더 유연하게 생각하는 연습이 필요하다.
    • 리뷰: 프레임이 끝났는지 여부(hasNext), 콘솔 출력값(Status)을 가진 필드를 없애고, 밖에서 메소드 불러서 해결하도록 하자. tryOne(), tryTwo()가 계속 이 필드 값들을 바꾸고 있는데 그 때문에 한 번에 여러 상태 값이 변함. 즉, 메소드가 여러가지 일을 하고 있는 것. 메소드를 쪼개야 재사용성이 높아진다.
      부모 클래스에서 필드를 private으로 못 쓸 경우, 자식 클래스로 필드를 내리거나 해당 필드를 쓰는 메소드 기능을 부모 클래스로 올려라. 자식 클래스들 메소드 내용이 다르면 공통되는 부분을 떼서 부모 클래스로 보낼 수 있는지 고민해라.
  7. Frame의 필드 2개 없앰. 해당 필드를 변경하고 가져오는 메소드 없앰. 그것을 대신한 메소드를 만들면서 연쇄적으로 많은 메소드 수정. 리팩토링.
    → 포비 말을 듣고 나서야 핀 변수를 가지고 프레임이 끝났는지 나타낼 수 있다는 걸 알았다. 은연중에 어떤 상태가 있으면 당연히 상태값 필드를 만들어야 한다고 생각했던 것 같다. 그게 제일 쉽게 느껴지기도 하고. 앞으로는 필드를 만들 때 꼭 필요한지 생각할 것 같다.
  8. codacy에 걸린 접근제한자 변경 - default 접근제한자를 사용하지 않도록 바꿈.

느낀점

전에 기능을 다 완성하고 클래스를 나눴더니 메소드들을 바꿔야 했던 기억이 있어서 0단계에서는 처음부터 클래스를 분류하려고 했다. 그런데 그때보다 복잡한 프로그램이라 클래스를 어떻게 나눠야할 지 확신이 서지 않았고 너무 많은 클래스를 만들어서 정리가 되지 않았다.
그런데 포비 조언대로 1단계부터는 방식을 바꿔 다 만들고 조금씩 수정했더니, 내가 어떤 클래스에서 무슨 일을 하는지 로직을 이해하면서 만드니까 부담스럽지 않았다. 그리고 5, 6단계에서 메소드 여러개를 수정했는데 내용이 1줄에서 5줄 사이인 메소드들이라 이 또한 부담을 줄여줬다. (수정을 할때 바꿀 부분이 적어서 그런 것 같은데 이 부분은 다음 스텝하면서 더 관찰하고 정리해야겠다.)
이 두가지 이유로 부담이 줄어드니까 코드 짜고 수정하는 게 재미있었다. 0단계에서는 며칠간 아무 진척도 없어서 우울했는데 1단계 이후부터는 속도도 나고 계속 조금씩 변하는 게 눈에 보이니까 더 나아갈 힘이 생겼다. 다만 stream, enum, 객체지향적 설계, 그 외 배운 것들 다 신경 안 쓰고 만들어서 그런 것들을 고려하면서 만드려면 어떻게 해야할 지가 고민이다. 그렇지만 일단 볼링게임에서는 최대한 단순하게 만들고 피드백 받으면 고치는 데 중점을 두고 싶다

그 외 리뷰

  • 부모 클래스에 내용 없는 메소드 쓸 거면 추상 클래스로 바꿔서 추상메소드 구현
  • 인스턴스 변수 안 쓰면 static 메소드로(이 부분은 다시 질문하기)
  • 변수명 일관되게, 이해하기 쉽게.
  • 사용하지 않는 테스트 코드는 지워라.
  • final static 변수는 클래스 바로 아래에 위치.
  • 클래스와 인스턴스 변수 사이에 공백 없어야 함.
  • toString()은 클래스 마지막에 위치.
  • 외부에서 안 쓰는 필드, 메소드는 private으로.

오늘 한 일

  1. 깃 브랜치 공부

    • HEAD: 현재 사용 중인 브랜치의 선두 부분
    • git branch [branchname] : 브랜치 생성
    • git branch : 브랜치 목록 전체 확인. ‘*’ 붙은 게 현재 선택된 브랜치
    • git checkout [branch] : 사용할 브랜치 지정
    • git checkout -b [branch] :브랜치 작성과 체크아웃을 한꺼번에 실행
    • 충돌이 안 났을 때 병합: git merge <병합할 브랜치> (fast-foward 병합)
    • 충돌나면 파일을 직접 수정한 뒤에 add, commit 함 (non fast-forward 병합)
    • 브랜치 종류
      1. master: 배포 가능한 상태만 관리
      1. develop: 통합 브랜치 역할. 평소에 이 브랜치를 기반으로 개발 진행
      1. feature: 토픽 브랜치 역할. 새로운 기능 개발 및 버그 수정이 필요할 때 분기. 개발 완료되면 develop 브랜치로 병합.

  2. 리눅스 명령어 공부

    • history | grep [검색할 내용]: 검색 내용을 가진 명령어 기록 찾아줌
    • 히스토리 저장 파일: home 디렉토리의 .bahs_history
    • vi 블록 지정하기: etc 후 v(비주얼 모드)
    • y: 블록 복사, p: 블록 붙여넣기
    • u: 실행 취소
    • U: 현재 커서가 위치하고 있는 줄에서 했던 모든 작업만 취소

오늘 한 일/배운 것

  • 리눅스 강의 수강
    내용: 리눅스 소개 + 간단한 명령어
  1. 커널(알맹이) = 운영체제 핵심
  2. 쉘(껍데기) = 사람과 연결
  3. bash shell = 기본. 서버에서 많이 씀.
  4. pwd 현재 디렉토리 확인
  5. home 디렉토리: 사용자의 기본 디렉토리(내 경우에는 home/사용자이름)
  6. cd 엔터: home 디렉토리로 감
  7. cd ~ : home 디렉토리 (‘~’ 는 홈 디렉토리를 뜻함.)
  8. 프로젝트 디렉토리는 home이 아니라 따로 워킹 디렉토리 만들고 그 안에서.
    홈 폴더에서 git init 하면 내 컴퓨터를 모두 공개하는 것..
  9. 윈도우는 C: 리눅스는 / 가 최상위.
  10. /부터 모든 경로를 적으면 절대경로. 그냥 적으면 상대경로.
  11. . 현재 디렉토리 .. 상위 디렉토리
    예: cd ../ ../ 상위의 상위 디렉토리로 이동.
    cd ././ 아무 일도 안 일어남.
  12. mkdir: make directory 상대, 절대경로 둘 다 사용 가능
  13. echo: 받은 문자열을 보여주거나 그 문자를 파일에 적을 수 있다.
    예: echo “some”> dir1/file.txt -> file.txt에 some이라는 글을 쓴다.
  14. cat: 파일 내용 보여줌.
  15. less: 긴 파일의 내용을 끊어서 표시
  16. q: 긴 내용 종료
  17. g: 처음으로
  18. history 후에 !히스토리 명령어 라인 넘버: 그 명령어 다시 보여줌
  • 깃 레벨1 수강
    내용: 기본 설명 및 필수 명령어
  1. 작업 디렉토리 = 워킹 트리. 직접 파일을 변경하는 곳.
  2. 프로젝트 기반으로 디렉토리를 만들기. 내문서나 home 쓰지 말고.
  3. 워킹트리에서 로컬 저장소(.git) 만들기
  • 저장소 삭제하고 싶으면 .git을 삭제하면 됨.
    • 방법 1. git clone 원격 저장소 주소
    • 방법 2. 디렉토리 만들고 그 디렉토리 안에서 git init
  1. 커밋하기 전에 add해서 스테이지(=인덱스)에 올려야 함.
  2. 커밋이 다 되서 워킹 트리와 로컬 저장소가 같은 내용이면 git status에서
    working tree clean 이라고 뜸.
  3. HEAD: 현재 작업 브랜치
    HEAD -> master 뜻: 내가 작업하는 곳이 마스터
  4. orgin: 원격 저장소
    git push origin master: 원격 저장소에서 master에 변경 내용을 보내라.
  5. pull: fetch + merge: 원격 저장소의 내용을 로컬 저장소에 복사하고,
    현재 작업 디렉토리까지 내용을 반영.
  • fetch: 원격 저장소와 로컬 저장소를 똑같이 만듦
  • merge: 로컬 저장소랑 워킹 디렉토리를 같게 만듦
  1. merge 병합: 서로 다른 커밋을 합치기 위해 사용.
  • git merge 브랜치 이름
  • git branch: 내가 어떤 브랜치에 있는지 보여줌.
  • git checkout: 브랜치를 바꿔줌.
  • 현업에서는 마스터에서 작업하지 않고, 브랜치에서 개발한 뒤 성공적으로 실행되면 마스터에 병합한다.
  • 깃 레벨2 수강
    내용: 브랜치 원리
  1. 깃에는 커밋 객체와 참조만 있다. 커밋하면 객체가 생기고, 이전 객체를 부모로 갖는다.
  2. 자식 객체는 부모 객체의 링크가 있어서 찾아갈 수 있다.
  3. 브랜치는 객체의 참조.
    그 외에는 이해하기 힘들었다.
